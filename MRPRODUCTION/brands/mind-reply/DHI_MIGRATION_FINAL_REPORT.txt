## Technical Implementation

### Build Stage Changes
```dockerfile
# OLD: FROM node:18-alpine
# NEW:
FROM dhi.io/node:22-alpine3.24-dev AS builder
```
- Includes npm and build tools
- Supports package installation
- Node 22 with stable Alpine 3.24

### Runtime Stage Changes
```dockerfile
# OLD: FROM node:18-alpine
# NEW:
FROM dhi.io/node:22-alpine3.24
```
- No shell (enhanced security)
- Minimal dependencies
- Non-root execution by default
- Suitable for application runtime only

### Multi-Stage Build Pattern
All three services maintain multi-stage architecture:
1. **Builder Stage:** Uses `-dev` image with npm
2. **Runtime Stage:** Uses minimal image for deployment

## Security Enhancements

| Feature | Benefit |
|---------|---------|
| **No Shell** | Cannot exec into container shells, reduced attack vector |
| **Non-Root User** | Prevents privilege escalation attacks |
| **Alpine Base** | ~5MB base image, fewer packages = fewer vulnerabilities |
| **Hardened Images** | Security-first design from Docker |
| **Dev/Runtime Split** | Build tools only in dev, removed from production |
| **Standard TLS** | Root CA certificates included by default |

## Support & Troubleshooting

### Issue: Cannot execute shell in container
**Expected Behavior:** Runtime images have no shell
**Solution:** Use node directly: `docker run image node -e "console.log('hello')"`

### Issue: npm ci fails during build
**Cause:** No package-lock.json present
**Solution:** Use `npm install` (implemented) or generate lock file: `npm install --package-lock-only`

### Issue: Build dependencies missing
**Solution:** Ensure build stage uses `-dev` image (implemented)

### Issue: Runtime fails to start
**Verify:** Ensure CMD uses executable format, not shell format
**Current Status:** All three services use proper exec format
